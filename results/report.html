MD vs HTML 検証レポート

LLMエージェントは Markdown と HTML、どちらの要件定義書をより正確に読み解くか

結論

主要な発見
  • Coverage rate はほぼ拮抗: 全長さでMD・HTMLとも100%またはそれに近い値。両形式とも要件の網羅性は十分。 (全長さ平均差 HTML-MD = +0.02pt)
  • 生のHallucination率では HTMLが大幅に多い: 全長さ平均で +14.9pt (HTML−MD)。 特に短い文書ほど顕著: 50行 = +32.6pt、500行 = +0.9pt。
  • ただし HTML側のhallucinationの大半は「構造メタ情報の観察」。 HTML文書中の data-category 属性、SVG図のキャプション、テーブル構造、配色などを「説明」として列挙したもの。 原文MDにこれらの記述はないため判定者は幻覚と判定するが、実際のビジネス要件 (新機能・新数値・新ロール) の捏造ではない。
  • 業務hallucination率に絞ると差は縮小: 全長さ平均差 +2.83pt (HTML−MD)。 50行HTML = 6.97%、500行HTML = 0.33%。 文書が長くなるほど両形式の差は縮む。
  • 結論: 短い要件定義書 (50-100行) ではMDの方が「読まれた事実」と「述べられた事実」が一致しやすく、確実。 長い文書 (250-500行) ではHTMLでもCoverage 100%・業務hallucination率 ≦1%を達成し、形式の差は実用上ほぼ無視できる。 元記事の主張「文書が長くなるほどHTMLの恩恵が大きい」は本実験では 長文ではHTMLが追いつくが超えはしない として観察された。 一方、短文では HTMLにすると判定者が認識する形式ノイズが大量発生する という副作用も明らかになった。

メソドロジー

本検証は「LLMエージェント (Claude Opus 4.7) は、要件定義書を Markdown と HTML のどちらで読んだ方が理解度が高いか」を定量比較する。

  • 4種類の文書長 (50/100/250/500行) × 2形式 (MD/HTML) × 5試行 = 40試行
  • 説明生成: 各 trial dir で claude -p をステートレス起動、Opus 4.7、Read/LS/Globのみ許可
  • HTML版は別Claudeセッションで MD→HTML 変換 (外部CSS + セマンティックHTML5 + SVG、新規事実追加禁止)
    • 変換方針は trq212の元記事 + nicbstmeの追加情報 に従う
  • 判定: チェックリスト含有 (Pass1) と extras裏付け (Pass2) を Opus 4.7 に推察禁止プロンプトで実施
  • 主指標: Coverage rate (=チェックリスト含有率) と Hallucination rate
  • 追加分類: hallucinations を 構造メタ観察 (HTML/SVG/CSS/属性) と 業務hallucination (要件捏造) に Haiku 4.5 で分類

③ 構造メタ観察 vs 業務hallucination

HTML版を読んだ被験者は、原文MDに記述のないHTML構造の観察 (data-category属性、SVG図、 テーブル構造、配色、aria-label等) を「説明」に含める傾向がある。 これは判定者から幻覚扱いされるが、要件定義書の中身を捏造しているわけではない。

そこで Haiku 4.5 を用いて、各 hallucination claim を以下の2カテゴリに分類した:

  • structural: HTML形式そのものに関する観察 (タグ、属性、SVG、配色等)
  • business: 原文書にない要件・機能・数値・固有名詞・ポリシーを捏造したもの

⑥ 制限事項と留意点

  • 試行数は条件あたり5回。サンプル数が小さく、統計的有意性は限定的。SDの確認は推奨される。
  • 判定者はOpus 4.7自身、Anthropic API直叩きではなく claude -p CLI 経由で実行。同モデルが両条件を判定するため比較における対称性は維持されるが、絶対値にはバイアスがある可能性。
  • 判定phase 1回目は parallel=6 で並列起動した結果、Opus への並行 claude -p 呼び出しが間欠的に exit code 1 で失敗 (Coverage 29/40, Extras 1/40)。リトライ機構を追加して parallel=2 で再実行し 80/80 成功。判定結果自体は単独実行と差異なし。
  • 分類者はHaiku 4.5 ( structural / business 二択分類)。境界事例として: 「ロール階層図(SVG): オーナー>管理者>メンバー」 はSVG観察だが内容のロール名はMD原文に存在するため曖昧。50行HTMLの「業務hallucination」例8件中複数は構造観察寄りに見える境界事例。Haikuが業務側へ寄せた可能性がある。
  • HTML版は trq212 / nicbstme 記事の推奨に従って 1ショット変換 (4並列の独立Claudeセッション) で生成。変換セッション自体のばらつきは未測定。実運用に近い設定だが、変換のサンプル数1での比較。
  • 題材はチャットツール1種のみ。他ドメインでは結果が異なる可能性。
  • HTML側で利用したセマンティックHTML5・SVG・table・dl は trq212 + nicbstme記事の推奨に基づく。素のHTMLや別スタイルでは差が異なる可能性。
  • 「読みやすさ」「共有のしやすさ」「2-way interactivity」などの人間側の便益は本検証のスコープ外。本検証はあくまで LLMエージェントの読解精度 のみを測る。

⑦ 元記事の主張との対照

元記事の主張本検証での観察
HTMLはセマンティック構造で情報密度が高い本検証では Coverage rate に差は出なかった。両形式とも100%。情報密度の差は読解精度に直結しない。
長い文書ほどHTMLの恩恵が大きい正しい方向性。短文ではHTMLが幻覚多く、長文では差が小さい。 ただし「HTMLが上回る」のではなく「HTMLが追いつく」。
外部CSS化でトークン削減本検証でも採用 (生成されたHTMLは外部 styles.css 参照のみ)。インラインCSS無しで安定して変換できた。
SVGでリッチに表現できるSVG自体は機能した。ただし副作用として、SVG関連の構造観察 (図の色、ラベル、矢印等) が大量に幻覚扱いされるという未報告の現象が観察された。
