# 100.md → requirements.html 変換ログ

## 全体方針

- セクション (1〜11) を `<section>` 単位でラップし、`aria-labelledby` で見出しと関連付け。
- 章番号付きの `<h2>`、サブグルーピングで `<h3>` を導入 (MDの本文が箇条書きで論点が並んでいる箇所のみ。新たな事実は追加していない)。
- 各要件項目には `data-category` 属性を付与し、章のテーマでカテゴリ分類。

## カテゴリ設計 (data-category)

| カテゴリ値 | 用途 |
| --- | --- |
| `user` | ユーザーアカウント関連 |
| `workspace` | ワークスペース定義 |
| `presence` | プレゼンス状態 |
| `channel` | チャネル仕様 |
| `message` | メッセージ仕様 |
| `mention` | メンション仕様 |
| `dm` | DM仕様 |
| `search` | 検索機能 |
| `notification` | 通知設定 |
| `nfr` | 非機能要件 |
| `tech` | 技術スタック |
| `security` | セキュリティ・機密 (DMの閲覧不可も含めた) |
| `warning` | 注意・制約 (最後の管理者退出時の権限委譲、`@channel`/`@here` 制限) |

## 各セクションへの構造マッピング

### 2. ユーザーとワークスペース
- 6項目を `<ul><li data-category>` で列挙。
- 末尾にプレゼンス状態遷移の **SVG figure** を1つ追加。MDで明示されている「3状態」「10分無操作で退席中に自動遷移」のみを視覚化。オフライン遷移条件はMDに無いので破線で「条件未明示」を示す表現に留め、新たな事実（ログオフ時遷移など）は付与していない。

### 3. チャネル
- パブリック/プライベートの2種別を `<table>` 化（種別×閲覧範囲）。
- 命名規則・参加・退出を `<ul>` で列挙。
- 「最後の管理者は権限委譲が必要」はネスト `<ul>` の子項目に置き、`data-category="warning"` 強調。

### 5. メンション
- `@ユーザー名` の通常メンションは `<ul>` 列挙。
- `@channel` / `@here` は **用語定義** とみなし `<dl>` で構造化。
- 制限規則は警告 `<p data-category="warning">` で分離。

### 7. 検索
- フィルタ3種を `<table>` 化（フィルタ表記 × 機能）。
- 並び替え・1000件ページングは別 `<h3>` グループ。

### 8. 通知
- 3レベル選択肢を `<table>` で表現（カラム1の単純表）。
- DM扱い・優先順位 (デフォルト/上書き) を `<h3>` で分離。
- 一時停止オプション (30分/1時間/2時間/翌朝9時) と「バッジは継続」は別 `<h3>`。

### 9. 非機能要件
- 6項目すべてを `<table>` で項目×要件の二列に整形 (NFRはタブラーデータと判断)。

### 10. 技術スタック制約
- アプリ層 (FE/BE) を `<dl>` で用語定義型に。
- データストア3種 (PostgreSQL/Redis/Elasticsearch) を `<table>` 化。
- 通信・インフラ・運用は再び `<dl>` でプロダクト/役割の対応をまとめた。
- 章内を3つの `<h3>` で再構成しているが、MD内の項目はすべて元の表現で保存。

### 11. セキュリティ要件
- 3項目を `<ul data-category="security">` で列挙。

## 判断に迷った箇所

1. **プレゼンスSVG**: MDには「10分無操作 → 退席中」しか遷移条件が無い。オフラインへの遷移条件は記載なしのため、矢印を破線にしてラベル無しで描画。発明を避けるため figcaption も「自動遷移」のみに留めた。
2. **章10 (技術スタック) のサブ構造化**: 元MDは見出し「データストア」のみ階層化されており、他はフラット。HTMLでは可読性のため 10.1/10.2/10.3 と便宜的に区切ったが、項目自体はすべて原文のまま。新規分類名 (アプリケーション層など) を加えたが、これは構造化ラベルであり要件追加ではないと判断。
3. **章8の通知レベル表**: 単一カラム表になるため `<ul>` でも良かったが、章内に他の `<ul>` があり視覚的に区別したかったため `<table>` を選択。
4. **`<details>` 不採用**: 100行レベルの短い文書で、折り畳むほどの低重要度補足が無いため `<details>/<summary>` は使わない判断。
5. **DMの「他人のDMは閲覧できない」**: 第6章の項目だが内容はセキュリティ性が強いので `data-category="security"` を付与した (章カテゴリより内容優先)。

- 全11章すべて反映。
- 数値: 1アカウント/メアド、80文字、4000文字、10分、500ユーザー、95p 500ms、95p 1秒/1000万メッセージ、1000件、30分/1時間/2時間/翌朝9時、24時間、bcrypt cost=12、TLS 1.2、React 18、Go 1.22、PostgreSQL 16、Redis 7、Elasticsearch 8、ハートビート30秒、RFC 6455、ap-northeast-1 — すべて記載確認済み。
- 例 (`acme.linkr.app`、`@ユーザー名`、`@channel`、`@here`、`from:`、`in:`、`before:`/`after:`) もすべて `<code>` で保持。
