# 250.md → requirements.html 変換ログ

## 全体方針

- ソースMDの章構成 (1〜21章) をそのまま `<section>` で再現し、各 section に `data-category` を付与。
  - 1章: `overview`
  - 2〜18章: 機能要件 → `functional`
  - 19章: `nfr`
  - 20章: `tech`
  - 21章: `security`
- 見出し階層は `h1` (タイトル) → `h2` (章) → `h3` (5.1/5.2/5.3 およびデータストア) を厳守。
- インラインコード (`@channel`, `acme.linkr.app`, `from:ユーザー名` 等) は `<code>` でマークアップ。

## 構造選択の判断

### テーブル化したもの
1. **3章 ロールと権限**: 「**オーナー**: 〜」のような太字+説明はタブラーなので `<table>` に変換 (caption 付与)。
2. **9章 検索フィルタ**: フィルタ構文 + 用途のペアを `<table>` 化。
3. **19章 非機能要件**: 項目 + 要件のペアテーブル。
4. **20章 技術スタック**: レイヤー + 構成のペアテーブル。

### 定義リスト (`<dl>`) を使ったもの
1. **4章 チャネル種別** (パブリック / プライベート): 用語と説明のペア。
2. **20章 データストア**: PostgreSQL/Redis/Elasticsearch/S3 を製品名+用途のペアとして `<dl>` で表現。
   - 技術スタックの「データストア」だけは元MDがネスト箇条書きでテーブル化しづらかったため `<dl>` を選択。

### 箇条書きを維持したもの
- 2章, 4〜8章, 10〜18章, 21章: 順序のないフラットな要件はそのまま `<ul>` で維持し、ネスト構造もそのまま入れ子に。
- 3章 末尾の運用ルール (オーナーの委譲・最低1名) は表に入れず `<ul>` で並列配置 (権限定義とは性質が異なるため)。

## SVG図

明示されている関係性のみ可視化:

1. **プレゼンス状態遷移図 (2章)**: MDに「オンライン / 退席中 / オフラインの3種類」「10分間操作がない場合に自動的に退席中」と明記されているため、3状態+10分タイマー遷移を矢印で描画。退席中→オフラインへの遷移条件はMDに無いため矢印のみで条件ラベルは付けず、無記述事実を追加しない。
2. **ロール階層図 (3章)**: MDに3ロール (オーナー/管理者/メンバー) と権限の優劣 (「オーナーへの昇格はオーナーのみ可能」「管理者はオーナー以外を付与可能」) が明記されているため、垂直階層図として表現。

新規発明的な図 (システム構成図、データフロー、シーケンス等) は MD に明記がないため作成しない。

## data 属性

- `data-category` を `<section>` レベルで付与: `overview`, `functional`, `nfr`, `tech`, `security`。
- styles.css の `.badge[data-category="..."]` 用カラーリング規約に整合させた値 (`security`, `tech`, `nfr`) を採用。

## 判断に迷った箇所

1. **5章「メッセージ」と 5.1〜5.3 のセクション関係**: 5章本体は箇条書きで5.1以下は子セクション。`<section>` を入れ子にし `h3` 子見出しで階層維持。
2. **3章ロール定義の表現**: 「**オーナー**: 〜」のように Markdown では太字+コロンで定義リスト的だが、行数が3つで横並びの権限比較が見やすいためテーブルを選択。
3. **20章データストア**: テーブル統一 vs `<dl>`。データストアのみネスト箇条書きで「製品+用途」の典型ペアだったため `<dl>` を選択 (テーブルにすると親テーブルとの2段階構造になり可読性が落ちる)。
4. **検索フィルタの `before:` / `after:`**: 元MDで同一行に併記されていたためテーブルでも1行にまとめて記載。同様に `has:link` / `has:file` / `has:reaction` も1行統合。
5. **プレゼンス図の遷移条件**: MDは「10分無操作で退席中へ」しか明記しないため、退席中→オフラインへの矢印は描いたがラベルは付与せず憶測を排除。

- 例として記載された `acme.linkr.app`, `@team-frontend` などのリテラルも `<code>` を付けて保存。
