# 50.md → requirements.html 変換ログ

## 全体方針

- ソースMD (7セクション・約50行) を完全反映。事実・数値の追加削除なし。
- 各セクションを `<section>` でラップし、`aria-labelledby` で見出しとひも付け。
- 要件項目には `data-category` を付与 (functional / security / nfr / tech / warning)。

## セクション別の判断

### 2. ユーザーとワークスペース
- 4項目は単純な箇条書きなので `<ul>` を維持。
- MDで明示されている「ユーザー ↔ 複数ワークスペース ↔ サブドメイン」関係を `<svg>` で1点だけ視覚化。新規関係は発明していない。

### 3. チャネル
- 「パブリック / プライベート」のネスト箇条書き (種別と閲覧範囲) は明確にタブラーなので `<table>` (caption付き、`<th scope="row">`) に変換。
- 残りの運用ルールは `<ul>` のまま (3.1 / 3.2 のサブ見出しでグルーピング)。
- 「最後の管理者は退出前に管理権限を委譲する必要がある」は警告的な前提条件なので `data-category="warning"`。

### 4. メッセージ / 5. DM
- 単純箇条書きで保持。
- DMの第三者非公開 (管理者でも閲覧不可) は機密性に関わるため `data-category="security"`。

### 6. 非機能要件
- 4項目すべて「項目: 値」のタブラー構造なので `<table>` 化、`data-category="nfr"`。

### 7. 技術スタック制約
- 「カテゴリ: 技術選定」の対応関係なので `<dl>/<dt>/<dd>` を選択 (テーブルでなく定義リスト)。
- 全項目に `data-category="tech"`。

## 利用したセマンティック要素

- `<main>`, `<header>`, `<section>` (7) , `<h1>/<h2>/<h3>`
- `<table>` (2: チャネル種別, NFR一覧) + `<caption>` + `<th scope>`
- `<dl>/<dt>/<dd>` (技術スタック)
- `<ul>` (ネスト含む)
- `<figure>/<figcaption>` + `<svg role="img">` (1図)
- `<code>` (サブドメイン例)

## SVG図

- 1点のみ: 「ユーザー ↔ ワークスペースA/B ↔ サブドメイン」関係図。
- ソースMDに明示されている「複数ワークスペース所属」「サブドメイン識別 (`acme.linkr.app`)」のみを表現。新規発明なし。

## 迷った箇所

1. **3.1 チャネル種別を表にするかリストのままにするか**: MDではネスト箇条書きだが、種別×閲覧範囲の対応関係が明確なので可読性優先で `<table>` 化。
2. **技術スタックを table か dl か**: 7章は「項目名 → 説明文」型でラベル定義の意味合いが強いので `<dl>` を採用。NFR (6章) は短い値の比較感が強いので `<table>` を採用、と使い分け。
3. **`<details>` の使用**: ソースが既に簡潔で折り畳むほど冗長な補足がないため未使用。
4. **SVGの粒度**: チャネル/メッセージ/DM の関係も図示候補だったが、MDでの明示度が低く「発明」のリスクがあるため、最も明確な「ユーザー-ワークスペース-サブドメイン」のみに絞った。
